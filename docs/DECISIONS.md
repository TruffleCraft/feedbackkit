# Architecture Decision Records

One paragraph per decision, including the rejected alternative. New resolutions flow here from [`decision` issues](../../issues?q=is%3Aissue+label%3Adecision).

## ADR-001 · 2026-07-13 · Name "feedbackkit" (working title, rename window until announce)

The repo is `TruffleCraft/feedbackkit`. The npm bare name and the GitHub user `feedbackkit` are taken externally (branding collision possible at a public announce); `feedbridge` would be free everywhere. Since v1 needs neither npm nor a domain (self-hosted, script tag), we keep the established working title — renaming is free until the announce and expensive afterward. Final naming: decision issue #9 before the P2 exit.

## ADR-002 · 2026-07-13 · Seed-JSON config in P1, full admin UI in P2 (UC1)

P1 is configured via a Zod-validated seed JSON (identical schema to the later admin config import — not a throwaway artifact); the SCTT migration becomes the P1 exit criterion (consolidation = primary goal first). Rejected: a minimal admin in the MVP (+3–4 PT, ADMIN_TOKEN surface on the critical path, operator population in P1 = 1). The admin experience (Michel's product vision) arrives complete and hardened in P2.

## ADR-003 · 2026-07-13 · P4 providers demand-gated; webhook sink as the universal interim (UC2)

GitLab→Jira→Trello start only once ≥1 external operator asks for a second provider. Until then a normalized JSON webhook sink (P2, ~1 PT) covers any tool via n8n/Zapier/Actions. Rejected: P4 as a fixed calendar commitment (12–17 PT of adapter maintenance before buyer clarity).

## ADR-004 · 2026-07-13 · wrangler.template.toml invariant (fork update path)

The repo commits only `wrangler.template.toml`; the real `wrangler.toml` (operator-specific database_id/bucket) is gitignored and generated by `pnpm deploy` from build variables. Rejected: a real wrangler.toml in the repo (would have broken "sync fork = conflict-free", since upstream touches the file regularly — cron triggers, compatibility_date).

## ADR-005 · 2026-07-13 · Extract-then-form instead of multi-turn chat (carried from the 2026-07-08 review)

ONE LLM call extracts against the per-type required fields; missing items are requested as pre-filled form fields; the second POST is deterministic (server re-validates, re-derives title — no client trust). Rejected: a multi-turn chat loop (turnToken protocol, unbounded LLM cost, biggest UX risk; stays in the backlog if data shows demand).

## ADR-006 · 2026-07-13 · R2 with app-level retention instead of BugDrop's orphan branch

Screenshots/attachments live in R2 (unguessable keys, per-project prefix, `assets` table with expires_at + cron delete + admin-authed delete endpoint). Rejected: BugDrop's orphan branch in the target repo (images in Git history are practically undeletable — GDPR; repo bloat; GitHub-only) and R2 lifecycle rules as the retention mechanism (bucket-level, not per-project controllable). Known residual risk: GitHub's camo proxy caches inline images — documented in SECURITY.md, editing the issue is part of the deletion process.

## ADR-007 · 2026-07-13 · OpenAI-compatible LLM client as an architecture invariant

One client (configurable baseUrl+model+apiKey) covers OpenRouter (default), LiteLLM, Ollama, vLLM and local models. Rejected: provider-specific LLM adapters (unnecessary surface; OpenAI-compatible is the de facto standard).

## ADR-008 · 2026-07-13 · Team-feedback hardening (origin cache, LLM structured output, webhook HMAC)

Three tightenings accepted from team feedback: **(a) Origin check** uses an anchored regex (`^https://…$`, dots escaped, wildcard `*` bounded to `[^./]+`) plus a bounded-TTL isolate cache of the allowlist (≤60 s, keyed on a config-version row) instead of a D1 read per feedback POST — reconciles the earlier "no isolate cache in P1" stance with hot-path throughput (60 s propagation is harmless for origin changes). **(b) LLM extraction** enforces structured output (Zod→JSON schema via response_format/function calling); schema enforcement is best-effort (local models in P5), the hard gate is server-side Zod validation; invalid JSON → immediate seamless fallback with no retry loop. **(c) Webhook sink** signs every POST with HMAC-SHA256 (per-project secret + timestamp against replay) from day 1. Rejected: unsigned webhooks (spoofable), prompt instruction alone without schema enforcement (LLMs hallucinate fields), a D1 read per request (bottleneck under traffic).

## ADR-009 · 2026-07-13 · P0 validation spike skipped

The spike was skipped on Michel's decision. Validating the core thesis shifts to the SCTT migration at the P1 exit (real feedback + the completion-funnel KPI = the same evidence on the real instance). Eval fixtures (10 anonymized SCTT feedbacks) and the p90 < 8 s target moved into the P1 LLM building block; the BugDrop fork check runs as the first P1 task. Deliberately accepted tradeoff: P1 builds on a premise not validated until P1 — both review models flagged the spike as the key de-risking step; the mitigation is watching the funnel closely from the first SCTT day. Rejected: the spike as a hard gate before P1 (Michel: SCTT as his own low-risk property yields the data anyway).

## ADR-010 · 2026-07-13 · Webhook signing scheme locked (HMAC-SHA256, Stripe-style header)

Concrete signing spec for the P2 webhook sink (full reference: docs/WEBHOOKS.md). **HMAC-SHA256** chosen over SHA-384: ecosystem standard (GitHub/Stripe/GitLab/n8n/Zapier verify it out of the box — zero friction), and SHA-384's 64-bit-word CPU edge is nanoseconds, dwarfed ~10,000× by JSON serialization + network latency in a Worker; 256 bits is ample for webhook signatures. Scheme: sign `` `${timestamp}.${rawBody}` `` over the **raw request body** (never a re-serialized object — key ordering/whitespace would break it), header `X-FeedbackKit-Signature: t=<epoch>,v1=<hex>`, 32-byte hex per-project secret in D1, 300 s replay window, receiver uses constant-time compare with a length guard. Two active secrets during rotation. Rejected: SHA-384 (integration friction), signing parsed JSON (non-deterministic), `===` hash compare (timing attack).

## ADR-011 · 2026-07-13 · Default LLM model + cost frame (resolves Q5)

Default `google/gemma-4-26b-a4b-it`, budget 200 calls/day. Recommendations are organized **by category** in [docs/MODELS.md](MODELS.md) (full verified table + method); the seed/QUICKSTART default is the "category 1" pick. Chosen from a live smoke test of FeedbackKit's exact request shape + the OpenRouter models API: extraction is not a reasoning task, so a mid-size model suffices; gemma-4-26b-a4b has vision (screenshots are a core input) **and** structured output, was the cheapest ($0.06/M) *and* fastest (~0.69 s) vision model tested, and — the deciding factor for a privacy-first tool — is **open-weights**, so the SAME model runs hosted (OpenRouter) or on-prem (ADR-007). That collapses the hosted default and the local/privacy default into one pick. **Vision is mandatory** (screenshots are always collected and are a primary input), so text-only models — including OpenAI's cheap/fast open-weights `gpt-oss-20b/120b` — are excluded regardless of price. Categories (all multimodal): fully-managed max-reliability = `gemini-2.5-flash-lite`/`gpt-5-nano`; paid quality step-up for messy input / screenshot OCR = `gemini-2.5-flash`/`gpt-4o-mini`; local/privacy = open-weights `gemma-4-26b-a4b`/`mistral-small-3.2-24b`; free/testing = `gemma-4-26b-a4b-it:free`. All hosted picks are <$0.001/feedback. Rejected "no default, force operator to choose": a sensible cheap default with a documented budget cap serves the ≤15 min TTHW goal better than a blank field. Model stays operator config — this is only the default. `structuredOutput` flag (default true; off for endpoints without strict-schema support, e.g. `gemma-4-31b-it:free`) added to the LLM client so the local-model story (ADR-007) works. NB: an earlier draft wrongly assumed the Gemma tiers were text-only, and first picked gemma-3-27b — corrected to gemma-4-26b-a4b (cheaper + faster) against verified data.

## ADR-012 · 2026-07-14 · Conversational single follow-up (partial reversal of ADR-005)

Live SCTT testing showed the extract-then-form follow-up (three rigid pre-filled required boxes + "noch N Angaben") felt like a form/interrogation — the exact friction the product set out to remove; the old SCTT widget users liked was a single freetext panel. New flow: **freetext first**, and when required info is missing the LLM asks **ONE natural, targeted follow-up question** (composed in POST-1's extraction call via a `followUpQuestion` field), answered in **freetext**; POST-2 does **one** re-extraction of that answer (text-only, budget-gated) and then **always creates** (single-shot — no second question; the answer is folded into the issue so it's never lost). Bounded at **2 LLM calls/feedback max**, both gated by the daily budget. Wire change: `FeedbackResponse` `need_fields` → `follow_up { question, extracted }`; `FeedbackPayload.fields` → `followUpText`. This partially reverses **ADR-005** (which rejected multi-turn chat on unbounded-cost + UX-risk grounds) but explicitly avoids the rejected part: it is a single bounded turn, not an open loop with a turnToken. Widget state `completing` → `asking`. Also reskinned the widget to the SCTT dark-navy/teal "expanding pill" look as the themeable default (light+dark). Rejected: keeping the N-box form (form friction), and full multi-turn chat (unbounded cost — still out per ADR-005).
